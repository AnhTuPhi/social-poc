The common shape

All three POCs solve the same class of problem: a few hot entities (a celebrity, a viral post, a hot counter key) generate disproportionate work, and the naive read/write path collapses. The fix is always to decouple the cheap hot path from the expensive path.

POCHot entityNaive failureDecoupling move
#23 FeedCelebrity (180M followers)1 post → 180M writespush for normals, pull for celebs (hybrid)
#24 NotifyPopular post50 likes → 50 buzzesbuffer + dedup + time-window batch
#25 CounterViral like counter1 row, all writers serializeshard writes / absorb in RAM, reconcile later

Each POC lets you run the naive version, watch it fail in the live metrics, then flip a strategy and watch it recover. Pick a tab above to walk the flow.

POC #23 · fan-out

News Feed Fan-Out

Hard problem: A celebrity post can't become 180M synchronous writes (push), but a normal read can't become a fan-in over everyone you follow (pull). Optimizing one end breaks the other.
Solution shape — Hybrid: decide per author. normal → PUSH fan out to a small follower set at post time. celebrity → PULL store once, merge into the reader's feed at read time. Threshold: followerCount ≥ 10,000.

Flow

Hybrid read does both at once: take the precomputed deque (from normal authors' pushes) and merge celebrity pulls, then sort + trim to 50.

Key tech by responsibility

ResponsibilityMechanism
Precomputed timeline (push target)Map<Long, Deque<FeedItem>>, capped at 50
Follower / following graphConcurrentHashMap.newKeySet()
Celebrity classificationfollowerCount ≥ CELEBRITY_THRESHOLD
Cost visibilityAtomicLong pushWrites / pullMerges / latency
Tech debt: fan-out is synchronous in-request (no queue); graph + feeds in one heap; no ranking (strict reverse-chron); threshold is one global constant.

POC #24 · dedup + batch

Notification Engine

Hard problem: 50 likes in 5 minutes must become one notification — while respecting quiet hours, multiple channels with per-user prefs, and collapsing true duplicates.
Solution shape: never deliver on ingest. buffer → dedup key → time-window batch → deliver per channel. A @Scheduled(1s) flusher delivers only batches older than the user's window.

Flow

Dedup key: LIKE/COMMENT/MENTION → :targetId, FOLLOW → :self, SYSTEM → random (never deduped).

How each sub-problem is solved

Sub-problemMechanism
Notification storm50 events → 1 dedup key → 1 batch → 1 delivery/channel
True duplicatessame key merge()s (bumps count, unions actors)
Timingflush only when firstSeen older than batchWindowMs
Do-not-disturbPUSH in quiet hours → SUPPRESSED_QUIET_HOURS (email/in-app still go)
Per-user controlPUT /prefs — channels, quiet window, batch window, live
Tech debt: suppressed pushes are dropped not deferred; flusher scans whole buffer each second (O(users×keys)); quiet hours use server timezone; no durability/retries/receipts.

POC #25 · hot-key contention

Sharded Counter

Hard problem: thousands of concurrent increments to one hot key serialize behind a single lock/row. Throughput collapses to single-writer speed on an otherwise idle box.
Solution shape — three strategies, load-tested with virtual threads: NAIVE_ROW_LOCK the anti-pattern (measures its own contention), SHARDED 32 AtomicLong shards, sum on read, REDIS_FLUSH hot in-RAM counter + @Scheduled(1s) write-behind.

Flow

Key tech by responsibility

ResponsibilityMechanism
Serialize (anti-pattern)ReentrantLock per key + busyMicros(5)
Detect contentionlock.isLocked() before acquire → counter
Scale writes32 AtomicLong shards, ThreadLocalRandom pick
Absorb hot / durable-ishhot AtomicLong + @Scheduled flush to persisted
Read-your-writes (hot)read = max(hot, persisted)
Prove under loadvirtual-thread pool + CountDownLatch start gate
Tech debt: "Redis"/"DB" are JVM AtomicLongs (single-process only); busyMicros understates real IO; REDIS_FLUSH loses ≤1s of increments on crash; SHARDED returns -1 on incr (must GET to sum).

What breaks when you scale out

Every POC is single-process and in-memory on purpose. The correctness you see is an artifact of one instance. Scale to --replicas=3 (or two VMs behind a load balancer) and most of it silently breaks.

Two sharp edges: (1) each pod has its own heap → reads miss writes from other pods; (2) @Scheduled fires on every replica → duplicated side-effects.

The two legitimate fixes (pick one per piece of state)

FixIdeaResult
Externalizemove state to a shared store (Redis / DB / Kafka log)pods stay symmetric & stateless
Partitionone owner pod per key (consistent hash / Kafka partition / sticky)state stays in-heap, never duplicated

The three rules it reduces to

  • State out of the heap, or one owner per key. Never replicate the process and keep authoritative state inside it.
  • A @Scheduled job in a replicated deployment runs N times. Guard every background job with partition-ownership or leader election.
  • Match routing to the aggregation boundary. If correctness needs all events for a key to meet, route them to meet — round-robin is only safe for stateless work.
